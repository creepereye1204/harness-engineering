# CONSTITUTION

## 규칙 우선순위

규칙이 충돌하면 아래 순서가 이긴다.

```
안전 > 정확 > 유지보수 > 간결
```

예: 간결하게 쓰면 입력 검증이 빠진다 → 검증을 추가한다.
예: 추상화하면 더 유지보수하기 좋다 → 스펙에 없으면 추가하지 않는다.

---

## 1. 주석 없음

주석을 작성하지 않는다. 이름이 설명한다.
이름으로 의도를 전달하지 못하면 이름을 고친다.

유일한 예외 (두 가지):
- 외부 제약(외부 API 계약, 특정 버그 우회)으로 코드만으로는 이유를 알 수 없는 경우
- `deleted_at IS NULL` 필터를 의도적으로 생략하는 쿼리 (이유를 한 줄로 기술)

---

## 2. 현대적 스타일만

판단 기준: **가독성**. 새 문법이라도 더 읽기 어려우면 쓰지 않는다.

### Java
| 금지 | 대신 |
|------|------|
| Raw type (`List`) | `List<T>` |
| `new Thread()` | `@Async` 또는 Virtual Thread |
| XML 기반 Bean 설정 | 애노테이션 기반 |
| `Optional.get()` | `orElseThrow()` / `orElse()` |
| Stream이 더 복잡한 경우의 Stream 강제 사용 | 명시적 for-loop |

스트림이 기본이다. 아래 경우에만 for-loop를 선택한다:
- 인덱스가 필요한 경우 (`for (int i = 0; ...)`)
- 중간에 체크드 예외가 발생하는 경우
- `break` / `continue`가 필요한 경우

그 외의 경우에 "읽기 어렵다"는 이유로 for-loop를 선택하면 안 된다.

### React (프론트엔드 작업 시 적용)
| 금지 | 대신 |
|------|------|
| 클래스 컴포넌트 | 함수형 컴포넌트 + hooks |
| `var` | `const` / `let` |
| Callback 중첩 | `async/await` |

불명확한 경우 판단 기준: 해당 언어/프레임워크 공식 문서의 현재 권장 방식.

---

## 3. 유지보수 가능한 구조

**함수**: 한 함수는 한 가지 일만 한다. 20줄을 넘으면 분리한다. 단, 순차적 단계가 하나의 흐름을 구성하는 경우(예: 입력 검증 → 변환 → 저장 파이프라인) 30줄까지 허용한다.
**중첩**: 3단 이상 if/try 중첩은 early return 또는 추출로 평탄화한다.
**중복**: 동일 로직이 3회 이상 반복되면 추출한다. 2회는 허용한다.
**추상화**: 현재 사용처가 2개 미만이면 추상화하지 않는다.

## 4. 네 가지 절대 규칙

**간결하게**
동작에 필요한 최소한만 작성한다.
방어 코드, 미래 대비 파라미터, 사용되지 않는 옵션은 추가하지 않는다.

**안전하게**
입력 검증은 시스템 경계(Controller, API 진입점)에서만 수행한다.
SQL 인젝션, XSS, IDOR, 인증 우회 — 생성 시점에 차단한다.
보안 패치가 즉시 필요하면 사용자 승인 없이 라이브러리를 추가할 수 있는 유일한 예외다.

**문서 없이 기능 추가 없음**
`docs/product-specs/`에 없는 기능은 구현하지 않는다.
스펙이 없거나 모호하면 구현 전 사용자에게 확인한다.
스펙에 버그가 있다고 판단되면 임의로 수정하지 않고 사용자에게 보고한다.

**테스트를 속이지 않는다**
`@Disabled`, 하드코딩된 반환값, 실제 동작을 우회하는 mock으로 테스트를 통과시키지 않는다.
테스트가 실패하면 먼저 원인을 파악한다.
- 코드 버그 → 코드를 수정한다
- 요구사항 변경 → 스펙 변경을 사용자에게 확인한 후, 테스트와 코드를 함께 수정한다

---

## 5. 테스트 정책

### 반드시 테스트해야 하는 것
- 핵심 비즈니스 로직: 카운터 동기화, 권한 체크, soft delete 필터, 인기글 점수 계산
- 경계 조건: 중복 투표, 본인 글 추천, 탈퇴 계정 이메일 재사용
- 보안 경계: 타인 리소스 수정/삭제 시도, 미인증 접근

### 테스트하지 않아도 되는 것
- 단순 getter/setter
- 프레임워크가 보장하는 동작 (Spring Security 필터 자체 등)

### 데이터 조회 API
단순 조회 API는 단위 테스트 대신 통합 테스트로 커버한다.
통합 테스트 환경: `@SpringBootTest` + Testcontainers MySQL.
H2는 MySQL 방언 차이(ON DUPLICATE KEY UPDATE, 특정 함수 등)로 거짓 통과가 발생하므로 사용하지 않는다.

### Mock 사용 원칙
외부 시스템(이메일 발송, 파일 서버)만 mock한다.
DB는 mock하지 않는다.

## 6. 워크플로우

### 작업 시작 전
1. `docs/exec-plans/active/`에서 현재 Phase와 작업 항목 확인
2. 해당 도메인의 `docs/product-specs/{도메인}.md` 정독
3. `docs/generated/db-schema.md`에서 관련 테이블 확인
4. 스펙에 없는 동작이 필요하면 구현 전 사용자에게 확인

### 백엔드 구현 순서
Flyway 마이그레이션 작성 → 사용자 확인 후 실행 → Entity → Repository → Service → Controller → Test

Flyway 수동 실행 명령:
```bash
# 로컬 (Docker 외부에서 실행 시)
./mvnw flyway:migrate

# Docker 환경 내에서 실행 시
docker compose exec backend ./mvnw flyway:migrate
```

### 프론트엔드 구현 순서
`docs/design-docs/`에서 컴포넌트/색상/타이포그래피 확인
→ 공통 컴포넌트 재사용 우선 → API 연동 → 주요 플로우 동작 확인

### 작업 완료 기준
exec-plan 해당 Phase의 **완료 기준** 항목이 모두 통과해야 완료다.
`docs/QUALITY_SCORE.md` 체크리스트 90% 이상 달성 필수.
완료 후 exec-plan의 체크박스를 업데이트한다.

### 의사결정 규칙

| 상황 | 행동 |
|---|---|
| 스펙이 모호하다 | 구현 전 사용자에게 질문 |
| 스펙에 버그가 있다 | 임의 수정 금지, 사용자에게 보고 |
| 라이브러리 추가가 필요하다 | 사용자 승인 후 진행 (보안 패치 예외) |
| DB 스키마 변경이 필요하다 | Flyway 파일 작성 → 실행 전 승인 요청 |
| 테스트가 실패한다 | 원인 파악 → 코드 버그면 코드 수정, 요구사항 변경이면 사용자 확인 후 함께 수정 |
| 규칙끼리 충돌한다 | 우선순위 순서 적용: 안전 > 정확 > 유지보수 > 간결 |
