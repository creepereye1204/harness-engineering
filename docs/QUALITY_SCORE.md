# docs/QUALITY_SCORE.md — 품질 기준

## 개요
Claude Code가 코드 작성 시 스스로 품질을 평가하는 기준.
PR 또는 작업 완료 전 아래 체크리스트를 확인할 것.

## 백엔드 체크리스트

### 기능
- [ ] product-specs의 요구사항을 모두 충족하는가
- [ ] 모든 예외 처리가 명세대로 구현되었는가
- [ ] API 응답 포맷이 ARCHITECTURE.md 기준을 따르는가
- [ ] 에러 코드가 {DOMAIN}_{ERROR_TYPE} 형식을 따르는가

### 코드 품질
- [ ] 비즈니스 로직이 Service 레이어에만 있는가
- [ ] Controller는 요청/응답 처리만 하는가
- [ ] 하드코딩된 값이 없는가 (운영 정책값은 service_settings 참조)
- [ ] N+1 쿼리가 발생하지 않는가

### 보안
- [ ] 인증/권한 체크가 누락되지 않았는가
- [ ] SQL Injection 방지 (JPA 파라미터 바인딩 사용)
- [ ] XSS 방지 (HTML 입력 sanitize 처리)
- [ ] 민감한 정보가 응답에 포함되지 않는가 (비밀번호 등)

### DB
- [ ] 스키마 변경이 Flyway 마이그레이션으로 관리되는가
- [ ] 목록 조회 쿼리에 deleted_at IS NULL 필터가 있는가
- [ ] 카운터 증감이 벌크 업데이트 쿼리로 처리되는가
- [ ] 인덱스가 필요한 컬럼에 적용되었는가 (`docs/generated/db-schema.md` 권장 인덱스 섹션 참조)

### 테스트
- [ ] 핵심 비즈니스 로직 단위 테스트 작성
- [ ] 주요 API 통합 테스트 작성 (User·Post·Comment·Vote·Board 도메인의 생성·수정·삭제 API 필수. 조회 API는 CONSTITUTION.md 테스트 정책 참조)

## 프론트엔드 체크리스트

### 기능
- [ ] 로딩 상태 처리가 되어 있는가 (Skeleton)
- [ ] 에러 상태 처리가 되어 있는가 (Toast)
- [ ] 빈 상태(empty state) UI가 있는가

### 코드 품질
- [ ] theme.* 없이 하드코딩된 색상/크기가 없는가
- [ ] 컴포넌트가 200줄을 넘지 않는가
- [ ] API 호출이 api/ 폴더에 분리되어 있는가
- [ ] 낙관적 업데이트 실패 시 롤백 처리가 있는가

### UX
- [ ] 모바일(375px)에서 정상 동작하는가
- [ ] 다크 모드에서 깨지는 UI가 없는가
- [ ] 버튼 중복 클릭 방지가 되어 있는가

## 점수 기준
- 체크리스트 90% 이상: 정상 진행
- 70~90%: 미충족 항목 이슈 등록 후 진행
- 70% 미만: 작업 중단 후 보완 우선
